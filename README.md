## Question 1 – Controlling load power over time on Resistive and Inductive loads

### Problem Summary

Controlling the power from a pure resistive load where the power cannot be instantly changed from 0% to 100%. Propose the changes to use this system on an inductive load instead.

### Solution

A suitable method is to control the load through PWM generated by the microcontroller and applied to a power stage such as a MOSFET. The average power delivered to the load is adjusted by varying the PWM duty cycle. The PWM frequency should be selected according to the characteristics of the load and switching device, balancing switching losses, efficiency and response time.

To avoid an abrupt transition from 0% to 100% power, the firmware should implement a ramp (soft-start) function, gradually increasing the duty cycle over time according to the application requirements.

For a purely resistive load, this method can be applied directly since the load responds to the average power delivered.

If the load is inductive (motors, relays, solenoids, etc.), additional precautions are required. When switching an inductive load, energy stored in the magnetic field can generate high voltage spikes when the current is interrupted. Therefore, protection circuits such as flyback diodes, snubbers, or transient suppression devices must be used to protect the switching device.

PWM can still be used for inductive loads, provided that the switching frequency and protection circuitry are properly designed.

## Question 2 – Button State Machine

### Problem Summary

Design a state machine for a button with three states: **OFF**, **ON**, and **PROTECTED**.

- **OFF** (default): button is not pressed.
- **ON**: button is currently being pressed.
- **PROTECTED**: button was just released; the system stays in this state for **10 seconds** before returning to OFF. If the button is pressed again during this window, the machine transitions back to ON immediately.

### Design Decisions

#### Non-blocking timing with `millis()`

A naive implementation might use `delay(10000)` or a `while` loop with a counter to wait for the 10-second window. Both approaches **block the processor entirely** — during that time, the microcontroller cannot respond to other inputs, update outputs, or run any other logic.

Instead, this implementation uses `millis()` to record a **timestamp** (`protectedStartTime`) at the moment the machine enters the PROTECTED state. On every iteration of `loop()`, the code checks whether the elapsed time has reached 10 seconds:

```c
(millis() - protectedStartTime) >= PROTECTED_TIME_MS
```

If not, the machine simply stays in PROTECTED and returns — the `loop()` will run again in microseconds and check again. This is the standard **non-blocking** pattern for timing in embedded systems.

#### Measuring Button Press Duration

Besides detecting state transitions, the implementation also measures how long the button remains pressed.

When the state machine enters the ON state, a timestamp is stored:

```c
pressStartTime = millis();
```

When the button is released and the state transitions from ON to PROTECTED, the elapsed time is calculated:

```c
pressDuration = millis() - pressStartTime;
```
This value represents the button press duration in milliseconds and is reported through the serial interface.

Measuring press duration is a common embedded-system requirement because different actions can be associated with short presses, long presses, or very long presses while still using the same state machine structure.

#### Hardware abstraction via `readButton()`

The low-level GPIO read is isolated in a single function:

```c
bool readButton(void) {
    return (digitalRead(BUTTON_PIN) == BUTTON_PRESSED_LEVEL);
}
```

To port this code to a different platform (STM32 HAL, ESP32, bare-metal AVR), only this function needs to change. The state machine logic above it remains untouched.

#### Button wiring

The button is wired **active-LOW** with the microcontroller's internal pull-up resistor enabled (`INPUT_PULLUP`). This is the most common and electrically safe configuration:
- No external resistor required.
- The pin reads HIGH when the button is open, LOW when pressed.

#### State machine structure

States are defined as a `typedef enum` for readability and type safety. Transitions are handled in a `switch/case` block — one `case` per state — which is the idiomatic C pattern for state machines:

```
OFF      → button pressed               → ON
ON       → button released              → PROTECTED (timestamp recorded)
PROTECTED → button pressed again        → ON
PROTECTED → 10 seconds elapsed, no press → OFF
```

## Question 3 – Cortex-M HardFault Debugging

### Problem Summary

Found the root cause from an intermittent system failure on a Cortex-M–based embedded device running bare-metal firmware, where:

- The system shows a hard fault after several hours of operation;
- During high interrupt load;
- No obvious stack overflow detected;
- The failure is not reproducible in a lab environment with standard test cases.

Explain, in detail, how:

- Systematically diagnose and root-cause this issue;
- Leverage:
    - hardware features (e.g., fault registers, stack frames);
    - firmware instrumentation;
    - interrupt priority analysis;
    - memory management considerations (e.g., stack/heap usage, linker configuration)
    - and any tools (e.g., JTAG/SWD, trace, logging)

### Solution

Since the failure is intermittent, occurs only after several hours of operation, and cannot be reproduced with standard laboratory tests, I would approach this problem using a systematic root-cause investigation process rather than immediately assuming a specific cause.

My first step would be to collect as much information as possible when the HardFault occurs. On a Cortex-M microcontroller, the processor provides fault registers and execution context information that can help identify what was happening when the fault occurred. I would capture this information and save it for later analysis, especially if the device automatically resets after the fault.

Next, I would determine which part of the firmware was executing at the time of the failure. Using the program counter information, debug symbols, and the firmware map file, I would identify the function or code section that was active when the HardFault happened. This helps narrow down the investigation significantly.

Since the problem is associated with high interrupt activity, I would carefully review the interrupt architecture. I would analyze interrupt priorities, nesting behavior, interrupt frequency, and any shared resources accessed by both interrupts and the main application. Problems caused by timing, unexpected interrupt interactions, or incorrect interrupt configuration become more likely under heavy interrupt load.

I would also investigate memory-related issues. Although the problem statement indicates that no obvious stack overflow has been detected, I would still verify stack usage under worst-case interrupt nesting conditions, since excessive interrupt activity can significantly increase stack consumption. I would also review memory allocation, buffer sizes, pointer usage, and linker configuration to identify possible memory corruption or invalid memory accesses.

To increase visibility into the system behavior, I would add diagnostic instrumentation. Examples include event logs, error counters, timestamps, and records of important system events. Since the fault cannot be reproduced using normal laboratory tests, collecting information from real operating conditions becomes especially important.

For debugging, I would use available tools such as SWD/JTAG interfaces to inspect processor registers, memory contents, and fault information. If hardware trace capabilities are available, I would also use them to better understand the sequence of events leading to the failure.

The main classes of problems I would investigate include memory corruption, invalid pointer accesses, incorrect interrupt configuration, excessive interrupt nesting, timing-related issues, and unexpected interactions between interrupts and the main application.

In summary, my strategy would be to first make the failure observable, then collect evidence, analyze the processor and firmware state at the moment of the fault, and systematically eliminate possible causes until the root cause is identified.

## Question 4 – UART Communication Protocol

### Problem Summary

Design a UART communication protocol that:
1. Guarantees each message is received correctly and its content is intact.
2. Uses **interrupts** (not polling) to receive incoming bytes.
3. Each message contains a **command** field and a **payload** whose format depends on the command.
4. Payload values may be **signed** (positive or negative).

### Frame Format

```
[START_BYTE] [CMD] [LEN] [PAYLOAD...] [CHECKSUM]
```

| Field       | Size     | Description                                      |
|-------------|----------|--------------------------------------------------|
| START_BYTE  | 1 byte   | Fixed value `0xAA` — marks the beginning of a frame |
| CMD         | 1 byte   | Command identifier (e.g., `0x01` = set speed)    |
| LEN         | 1 byte   | Number of payload bytes that follow               |
| PAYLOAD     | LEN bytes| Data bytes; interpreted according to CMD          |
| CHECKSUM    | 1 byte   | XOR of CMD, LEN, and all payload bytes            |

#### Why this format ensures correctness

- **START_BYTE** allows the receiver to re-synchronize after noise or partial frames — it discards bytes until it finds `0xAA`.
- **LEN** tells the receiver exactly how many bytes to collect before looking for the checksum, preventing misalignment.
- **CHECKSUM** (XOR) allows the receiver to detect corrupted frames. If the computed checksum does not match the received one, the frame is silently discarded and the parser resets. For higher reliability, a CRC-16 could replace the XOR checksum.

#### Transmission Reliability

The checksum guarantees message integrity but does not guarantee message delivery.

In applications requiring higher reliability, an ACK/NACK mechanism could be implemented:

- Receiver validates the frame.
- Receiver sends ACK if valid.
- Receiver sends NACK (or remains silent) if invalid.
- Transmitter retries after a timeout.

This approach is commonly used in industrial communication systems when message loss cannot be tolerated.

### Architecture: ISR + Ring Buffer + Parser

The reception is split into two layers:

#### Layer 1 – ISR (Interrupt Service Routine)

The hardware UART fires an interrupt on every received byte. The ISR does **one thing only**: push the byte into a circular ring buffer.

```c
ISR(USART_RX_vect) {
    uint8_t receivedByte = UDR0;
    ringBuffer_push(receivedByte);
}
```

ISRs must be as short as possible. Parsing, printing, or any blocking operation inside an ISR would delay other interrupts and destabilize the system.

#### Layer 2 – Parser (main loop)

`processUART()` is called from `loop()`. It drains the ring buffer byte by byte and runs a **state machine parser** that reassembles frames:

```
WAIT_START → CMD → LEN → PAYLOAD → CHECKSUM
```

Because it processes one byte per state transition and returns immediately if the buffer is empty, it is **fully non-blocking**. The rest of the application runs freely between calls.

#### Why a ring buffer?

The ISR and the main loop run asynchronously. The ring buffer decouples them safely:
- The ISR writes to the **head** pointer.
- The main loop reads from the **tail** pointer.
- No byte is lost as long as the main loop drains the buffer faster than it fills up.

### Signed Values

Payload values that represent quantities such as speed or position may be negative. In C, negative integers are stored in **two's complement** representation, which is the default for signed types (`int8_t`, `int16_t`, `int32_t`).

To reconstruct a signed 16-bit value from two received bytes (big-endian):

```c
int16_t speed = (int16_t)((uint16_t)payload[0] << 8 | payload[1]);
```

The explicit cast to `int16_t` tells the compiler to interpret the most significant bit as a sign bit. No manual bit manipulation is required — the CPU handles two's complement natively.

### Command Dispatch

After a valid frame passes the checksum check, `dispatchCommand()` uses a `switch/case` on the CMD field to extract and apply the payload:

| CMD  | Name             | Payload type | Range                        |
|------|------------------|--------------|------------------------------|
| 0x01 | CMD_SET_SPEED    | `int16_t`    | -32768 to +32767 RPM         |
| 0x02 | CMD_SET_POSITION | `int32_t`    | -(2³¹) to +(2³¹-1) counts   |
| 0x03 | CMD_SET_GAIN     | `int8_t`     | -128 to +127                 |

### Prior Experience

This architecture is directly informed by prior professional work: during a P&D role at Xmobots Aeroespacial, I independently reverse-engineered a proprietary UDP communication protocol for a UAV gimbal system — mapping a 68-byte packet structure with no documentation available — and developed a Python telemetry parser to decode and validate the data in real time. The same separation of concerns (raw byte collection → frame reassembly → payload interpretation) applied there as in this solution.

## Question 5

### Difference between the two encircled circuits

Both circuits condition a mechanical switch signal before delivering it to a microcontroller input pin. While they share the same general purpose, they differ in their logic polarity, pull configuration, and filtering complexity.

**Red encircled circuit — "Extra Switch" (pin PTD0/2, connector J11/SW1):**

A pull-up resistor R41 (1 kΩ) connects the +5 VDC rail to the switch node. Switch SW1 in connector J11 connects that node to GND when closed. The signal then passes through R44 (1 kΩ) in series to the microcontroller pin PTD0. A capacitor C26 (10 nF) and a transient voltage suppressor TVS (TRSB8/SMAJ7.0A) are placed between that series node and GND.

- **Logic:** active-LOW. Switch open → pin reads HIGH. Switch closed → pin pulled to GND → pin reads LOW.
- **R41 + R44:** form a current-limiting series path; R44 also works with C26 as a low-pass RC filter.
- **C26 (10 nF):** hardware low-pass filter to attenuate high-frequency noise and contact bounce.
- **TVS:** clamps voltage transients that could exceed the microcontroller's input voltage rating, protecting the pin from electrostatic discharge or inductive spikes.

**Green encircled circuit — "Door Switch" (pin PTF5/ADC0_SE13/29, connector J8/SW2):**

This circuit is more elaborate, reflecting the harsher electrical environment of a door switch in an appliance (proximity to motor, mechanical vibration). A pull-up resistor R45 (1 kΩ) connects +5 VDC to the switch node. The resistor network formed by R45 and R48 establishes the logic state seen by the microcontroller depending on the switch position. (the switch pulls the line high). R46 (1 kΩ) and R47 (680 Ω) form a resistive divider. C27 (220 pF) and C28 (10 nF) provide two-stage hardware filtering. R48 (3.3 kΩ) anchors the node toward GND when the switch is open. A TVS (TRSB9/SMAJ7.0A) provides transient protection.

- **Logic:** active-HIGH. Switch open → node pulled low via R48 → pin reads LOW. Switch closed → node pulled to +5 V via R45 → pin reads HIGH.
- **R46 + C27 and R47 + C28:** two RC filter stages providing more aggressive hardware filtering than the red circuit — appropriate for a door switch exposed to motor-induced noise.
- **Resistive divider (R45, R46, R47, R48):** also serves to limit the voltage and current seen by the microcontroller pin.
- **TVS:** same transient protection role as in the red circuit.

**Summary of differences:**

| | Red circuit (pin 2) | Green circuit (pin 29) |
|---|---|---|
| Switch logic | Active-LOW | Active-HIGH |
| Resting state | HIGH | LOW |
| Pull configuration | Pull-up (R41 to +5V) | Pull-up (R45) + pull-down (R48) |
| Hardware filtering | Single RC stage (R44 + C26) | Two RC stages (R46+C27, R47+C28) |
| Protection | TVS (SMAJ7.0A) | TVS (SMAJ7.0A) |
| Application | Extra switch (less noisy) | Door switch (noisier environment) |

### Microcontroller pin configuration

For both circuits, all bias and filtering components are external. The microcontroller pins must be configured as plain digital inputs with no internal pull resistors activated — the external network already defines the resting voltage level.

```c
pinMode(2,  INPUT);   // PTD0  — red circuit, active-LOW, external pull-up via R41
pinMode(29, INPUT);   // PTF5  — green circuit, active-HIGH, external network defines resting LOW
```

### Software filtering — debounce

Even with the hardware RC filters present, mechanical switches still produce contact bounce — rapid transitions over a few milliseconds as the contacts settle. The hardware filter attenuates high-frequency noise but does not eliminate bounce entirely. A software debounce layer provides the final protection against false readings.

The standard software solution is **time-based debouncing using millis()**. The firmware accepts a new switch state only after the signal has remained stable for a defined period (typically 10–50 ms). Any transition shorter than this threshold is discarded as bounce.

This is implemented as a non-blocking check inside the main loop: the time of the last detected transition is recorded, and on each loop iteration the firmware checks whether enough time has elapsed since the last change. Only then is the new state accepted as valid. This approach does not use delay() or any blocking construct, so the rest of the application continues running normally during the debounce window.

The additional filtering stage is particularly useful in appliance environments where motors, relays and switching power electronics may generate electromagnetic interference (EMI).

### Design document — switch reading from the green encircled circuit (pin 29)

**1. Hardware configuration**
- Switch SW2 connected to connector J8; closes to +5 VDC via pull-up R45 (1 kΩ).
- R48 (3.3 kΩ) holds the node low when the switch is open.
- Two-stage RC filter: R46 (1 kΩ) + C27 (220 pF) and R47 (680 Ω) + C28 (10 nF).
- TVS (SMAJ7.0A) for transient voltage protection.
- Pin PTF5/29 configured as: GPIO digital input, no internal pull activated.
- Resting state (switch open): pin reads LOW (logic 0).
- Active state (switch closed): pin reads HIGH (logic 1).

**2. Sampling method**
- Polling inside the main application loop.
- Interrupt-based reading is possible, but for a door switch polling is generally preferred due to its simplicity and because the signal changes occur at human timescales.

**3. Debounce algorithm**
- Time-based, non-blocking, using millis() for elapsed-time measurement.
- Debounce threshold: 20 ms (configurable via a named constant; may be reduced given the hardware RC filter already present).
- On each loop iteration:
  - Read the current raw pin state.
  - If the raw state differs from the last recorded reading, reset the stability timer.
  - If the raw state has remained unchanged for ≥ 20 ms, accept it as the stable state.
- Return the last accepted stable state to the application.

**4. Output interface**
- A boolean value: true = switch closed (door closed), false = switch open (door open).
- The rest of the application uses only this debounced value, never the raw pin reading.

**5. Assumptions and constraints**
- Hardware filter components are assumed to be correctly populated as shown in the schematic.
- The microcontroller input is assumed to be 5 V tolerant or protected by the external conditioning network shown in the schematic.
- The combined hardware + software filtering approach is intentional: hardware reduces the burden on software debounce; software eliminates any residual bounce the hardware does not fully suppress.

## Question 6

### Sensorless speed control of a permanent magnet synchronous motor (PMSM)

Controlling the speed of a PMSM without a physical sensor on the shaft — a technique known as **sensorless control** — eliminates the cost, weight, and potential failure point of a mechanical encoder or resolver. This is especially relevant in applications such as home appliances, where cost and reliability are critical.

The standard approach for high-performance sensorless speed control is **Field Oriented Control (FOC)** combined with a **rotor position and speed estimator**. FOC works in a rotating reference frame (the d-q frame) that is synchronized with the rotor magnetic flux, allowing independent control of torque-producing current (Iq) and flux-producing current (Id) using simple PI controllers.

### Block diagram

```
Speed setpoint (ω*)
        │
        ▼
  [Speed PI controller]
        │ Iq* (torque current reference)
        ▼
  [Current PI controllers — d/q frame]
   Id* → PI_d → Vd*
   Iq* → PI_q → Vq*
        │
        ▼
  [Inverse Park transform (d-q → α-β)]
  Uses estimated rotor angle θ̂
        │
        ▼
  [Inverse Clarke transform (α-β → three-phase a-b-c)]
        │
        ▼
  [PWM modulation (e.g. SVPWM)]
        │
        ▼
  [Three-phase inverter (IGBTs / MOSFETs)]
        │
        ▼
  [PMSM — three-phase windings]
        │
        ▼
  [Phase current sensors (Ia, Ib)]
        │
        ▼
  [Clarke transform (a-b-c → α-β)]
        │
        ▼
  [Park transform (α-β → d-q, using θ̂)]
        │ Id (measured), Iq (measured) — fed back to current PIs
        │
        ▼
  [Sensorless position / speed estimator]
  (e.g. Back-EMF observer or Sliding Mode Observer)
        │ θ̂ (estimated angle), ω̂ (estimated speed)
        └──────────────────────────────────────────────┐
                                                       ▼
                                          Fed back to Park/inverse Park
                                          and to the speed PI controller
```

### Explanation of each block

**Speed PI controller:** compares the desired speed (ω*) with the estimated speed (ω̂) and outputs the torque current reference (Iq*). The flux reference (Id*) is typically kept at zero for surface-mount PMSMs to maximize efficiency.

**Current PI controllers (d-q frame):** because the Park transform converts the sinusoidal phase currents into DC quantities in the d-q frame, simple PI controllers can regulate them with zero steady-state error. Vd* and Vq* are the voltage references that drive the motor.

**Inverse Park and Clarke transforms:** convert the voltage references from the rotating d-q frame back to three-phase quantities that can be applied to the inverter. These transforms require the estimated rotor angle θ̂.

**PWM modulation:** translates the three-phase voltage references into gate drive signals for the inverter switches. Space Vector PWM (SVPWM) is commonly used because it maximizes DC bus utilization and reduces harmonic content.

**Three-phase inverter:** converts the DC bus voltage into three-phase AC voltages applied to the motor windings, under the control of the PWM signals.

**Phase current sensors:** measure two of the three phase currents (the third can be derived from Kirchhoff's current law). These measurements are the primary feedback signals for the entire control loop.

**Clarke and Park transforms (feedback path):** convert the measured phase currents back into the d-q frame so the current PI controllers can compute the error and respond accordingly.

**Sensorless position and speed estimator:** this is the key block that replaces the physical sensor. As the rotor spins, it generates a Back-EMF (BEMF) voltage in the stator windings. This BEMF contains information about the rotor's position and speed. The estimator uses the measured currents and the applied voltages to compute a model of the motor and extract the rotor angle (θ̂) and speed (ω̂) from the difference between the model prediction and the actual measurements.

Common estimator implementations include:
- **Back-EMF observer**: reconstructs the BEMF from the motor electrical equations and extracts angle and speed from it. Reliable at medium-to-high speeds but less effective near zero speed. One limitation of Back-EMF based methods is that the generated voltage becomes very small at low speeds, making rotor position estimation more difficult during startup and near-zero speed operation.
- **Sliding Mode Observer (SMO)**: a robust nonlinear observer that is less sensitive to motor parameter variations. Widely used in industrial and appliance applications.
- **Extended Kalman Filter (EKF)**: provides optimal estimation in the presence of noise, at the cost of higher computational complexity.

### Key advantage of FOC over scalar (V/f) control

Scalar control simply adjusts the ratio of voltage to frequency applied to the motor. It is simple but provides poor dynamic response and limited torque control at low speeds. FOC, by controlling torque and flux independently in the d-q frame, delivers fast dynamic response, accurate torque control across the full speed range, and higher efficiency — all of which are required in demanding applications such as washing machine drums that must accelerate under load or maintain precise speed during spin cycles.

Therefore, sensorless FOC provides precise speed control, high efficiency and reduced system cost by eliminating the mechanical speed sensor while maintaining excellent dynamic performance.

At startup and very low speeds, sensorless estimators may require dedicated startup strategies because the generated Back-EMF is too small to provide reliable position information.
